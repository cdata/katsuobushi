## What to build

Found while dogfooding `sandbox dispatch`: a dispatched agent ended its turn **unreported and without committing/pushing**, so no work landed — but nothing surfaced that. `sandbox fetch` returned a branch containing only the `git stash` seed commits (`WIP on …` / `index on …`), and the orchestrator had to notice that by hand. Dispatch/the report bridge should make "no work landed" impossible to miss.

## Acceptance criteria

- [ ] `sandbox fetch` (or a bridge/status capability) detects when a fetched sandbox branch has no real commit beyond the seed and reports it clearly (distinct from a genuine `done`)
- [ ] the dispatched-agent liveness `ended-unreported` is surfaced as an actionable state, not just a log line
- [ ] documented recovery (re-prompt to commit/push/report, else reset + re-dispatch fresh) — cross-check the project-orchestration skill gotcha


## Review notes

- 2026-07-18 (sandbox reviewer, needs-changes, non-blocking): `is_seed_subject`
  only catches the **dirty-tree** stash seed (`WIP on …`). `resolve_seed` falls
  back to **HEAD** on a clean tree and `Seed::Resume` uses the prior branch
  commit — both real commits — so a clean-seed dispatch whose agent ends without
  committing yields a real-commit tip and is NOT flagged. The `fetch.rs` comment
  overclaims. Reliable fix: persist the seed SHA (computed in `resolve_seed`)
  and compare `tip != seed`. Also: the `index on ` arm is dead for the tip; the
  `branch_has_work` probe-failure fallback is untested.
- Response: softened the comment to its real scope, dropped the dead `index on`
  arm, added the fallback test. The robust seed-SHA comparison is filed as a
  follow-up (larger: threads the seed SHA through instance.json + fetch).
