When moving issues into the 'Ready' state, they should be sorted from top to bottom in the suggested acceptance order so that we can get a roughly linear chronology of what is landing.

## Review notes

Peer-reviewed in sandbox `review-5eb75c` — **NEEDS-CHANGES**. The reviewer
confirmed the sort semantics (topological deps first, then `created` ascending —
do NOT make created primary), but flagged that re-sorting the whole Ready lane on
every entry silently discards a manual `prioritize` order (conflicts with the
"position = priority" model). Product-owner decision (asked): **only place new
entrants** — insert each card entering Ready at its acceptance-order slot and
leave the incumbents untouched. Reworked accordingly: `acceptance_order` +
`Board::reorder_lane` removed; replaced by pure `select::ready_insertion_index`
(dependencies as hard bounds, `created` within the window) wired into
`set_status`, covered by 7 unit tests + an integration test. Two nits fixed
(`lint`-forbids-cycles overclaim; `zzzz`/`ffffff` test comment).

Re-reviewed after the rework in sandbox `review-5eb75c2` — **ACCEPT**. Verified
`ready_insertion_index` bounds are off-by-one-free (blocker mid-lane, blocks
mid-lane, dependency-violating manual order), incumbents left unmoved (single
`board.reorder`), index→Anchor mapping correct, old whole-lane sort fully
removed. `cargo test` 321 passed; clippy `-D warnings` green. (Also fixed an
out-of-scope pre-existing `Board::reorder` doc-comment drift.)
